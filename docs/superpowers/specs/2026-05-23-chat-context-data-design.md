# 老人端对话数据接入设计

## 背景

档案与记忆导入链路已经具备基础能力：前端可智能解析家庭资料，后端可通过 `/api/import` 保存到本地 SQLite。下一步应优先确认老人端对话会读取这些已保存数据并影响回复，而不是先进入云存储或声音克隆。

## 目标

老人端每次对话时，回复生成流程要使用现有本地档案库中的数据：

- 老人画像：姓名、性格、偏好、健康备注、说话特点、人生经历。
- AI 扮演角色：角色名、关系、称呼、性格、说话风格、陪伴方式。
- 家人档案：家人姓名、关系、偏好、习惯、备注。
- 家庭记忆：与当前话题或提及人物相关的共同记忆。

成功标准：导入家庭资料后，老人端回复能自然使用称呼、关系、老人偏好和共同记忆，不出现“根据资料”“系统显示”等元描述。

## 当前实现判断

`productization/chat_service.py` 已经具备主要数据接入路径：

1. 调用 `init_db()` 初始化本地库。
2. 通过 `load_all_personas()`、`load_all_memories()`、`load_all_family_profiles()`、`load_elder()` 读取已保存数据。
3. 用意图情绪识别结果中的 `talk_to` 和 `mentioned` 选择角色、过滤记忆和构建上下文。
4. 将老人画像、家人档案、记忆上下文传给 `build_response_system()`。
5. 在 debug 中返回 `top_memories`、`mentioned`、`talk_to` 和评分信息，便于验证。

因此本阶段不需要重写对话生成架构，重点是补强验证和前端可观测性。

## 推荐方案

采用“本地数据闭环优先”的方案：

1. 保持当前 SQLite 数据源，老人端对话继续读取本地 `companion.db`。
2. 增加一条端到端测试：先通过导入接口保存老人画像、角色、家人档案和记忆，再调用聊天接口，断言生成 prompt 包含这些数据。
3. 在老人端页面保留或补充简洁错误提示，确保后端不可用、LLM 调用失败时不会让页面崩溃。
4. 暂不接云存储和声音克隆。等“资料导入 -> 老人对话使用资料 -> 回复质量可验证”闭环稳定后，再迁移存储层和接语音能力。

## 数据流

1. 家属在档案与记忆页粘贴资料。
2. 前端调用 `/api/parse` 生成可编辑草稿。
3. 家属点击“一键保存”，前端调用 `/api/import`。
4. 后端写入 SQLite。
5. 老人在老人端输入一句话。
6. 前端调用 `/api/chat`。
7. 后端读取 SQLite 中的角色、老人画像、家人档案和记忆。
8. 对话服务识别意图、情绪、提及人物和对话对象。
9. 对话服务筛选并评分相关记忆，构建 prompt。
10. LLM 生成回复，经过老年适配和安全检查后返回。

## 错误处理

- 导入数据为空：聊天服务仍可用，用默认角色和无记忆上下文回复。
- LLM 意图识别失败：降级为“日常闲聊 / 平静 / 陪伴者”。
- LLM 回复生成失败：返回温和的重试提示。
- 保存接口未加载：前端提示重启 API 服务。

## 测试策略

- 保留现有 `test_chat_service_generates_reply_from_sqlite_data`，它验证对话服务会读取 SQLite 里的老人画像和记忆。
- 增加或强化 API 层测试：通过 `/api/import` 写入资料，再通过 `/api/chat` 调用聊天，断言 prompt 中包含导入内容。
- 运行全量 pytest 和前端 typecheck，确认导入、聊天和产品化基础测试不回退。

## 非目标

- 本阶段不迁移 Supabase。
- 本阶段不实现多家庭云端隔离。
- 本阶段不接真实声音克隆服务。
- 本阶段不改动真实 API Key 配置方式。
